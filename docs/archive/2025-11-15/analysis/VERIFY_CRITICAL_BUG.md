# Verification of Critical Bug: Source References

## The Issue

Source references are extracted from `research["subsolutions"][i]["synthesis"]`, but the response text is generated by the LLM and may be completely different.

## Code Flow Analysis

1. **Research produces synthesis** (`src/bop/orchestrator.py:294-309`):
   - Creates `subsolutions` with `synthesis` field
   - Synthesis is LLM-generated summary of tool results

2. **Final synthesis** (`src/bop/orchestrator.py:311-315`):
   - LLM synthesizes all subsolutions into `final_synthesis`
   - This is used for structured tier, not response text

3. **Response text generation** (`src/bop/agent.py:169-174`):
   - `_generate_response` creates the actual response
   - This is LLM-generated based on message + context
   - **May be completely different from synthesis!**

4. **Source references** (`src/bop/agent.py:177-180, 684-740`):
   - Matches sources to `synthesis` sentences
   - But response text is different!
   - **Sources cited don't match actual response claims**

## Example Scenario

**Research Synthesis**:
- "Trust is important. Systems need reliability. Knowledge requires verification."

**LLM Response Text**:
- "Trust serves as a foundational mechanism in knowledge systems, enabling reliable information exchange and reducing epistemic uncertainty through verification processes."

**Source References**:
- Matches sources to synthesis sentences ("Trust is important", etc.)
- But response text has different wording!
- Sources are attributed to synthesis, not response

## Impact

**CRITICAL**: Users see source citations, but those sources may not actually support the claims in the response text they're reading.

## Fix Options

1. **Extract claims from response text** and match to sources
2. **Use semantic similarity** to match response sentences to synthesis sentences  
3. **Include source info in LLM prompt** so response naturally includes citations
4. **Match sources to both** response text AND synthesis, show intersection
